# AI 에이전트로 하루 600커밋 하는 개발자의 비밀

## 영상 개요
- 주제: Peter Steinberger의 AI 에이전트 코딩 방법론과 실전 노하우
- 타겟: 1인 창업가, AI 자동화에 관심 있는 개발자, 생산성 향상을 원하는 IT 종사자
- 핵심 메시지: AI와 함께 일하는 새로운 방식 - Loop 닫기, 병렬 에이전트 운영, Builder로의 정체성 전환

### [0:30] Peter 소개

[B-roll: Peter Steinberger 사진]

Peter Steinberger는 PSPDF Kit이라는 PDF 프레임워크를 만든 사람이에요.
[자막: "PSPDF Kit - 10억+ 디바이스에서 사용"]

### [0:45] 엄청난 성공
이 프레임워크가 얼마나 많이 쓰이냐면, 10억 개 이상의 디바이스에서 사용되고 있어요.
여러분이 아이폰이나 아이패드에서 PDF 볼 때, 뒤에서 돌아가는 기술이 바로 이거예요.

13년 동안 이 회사를 운영했어요.

### [1:00] 번아웃과 3년 휴식
근데 이 사람, 완전히 번아웃이 왔대요.

그래서 뭘 했냐면, 3년 동안 기술에서 완전히 손을 뗐어요.
코딩도 안 하고, 기술 뉴스도 안 보고, 완전히 쉬었어요.

### [1:15] 2024년 복귀
그러다가 2024년에 복귀했는데, 세상이 완전히 달라져 있었대요.

심지어 에이전트가 스스로 테스트하고 디버깅까지 하는 시대가 온 거죠.

### [1:30] Claudebot 프로젝트

## 섹션 2: 비밀 1 - "Loop 닫기" 원칙 [2:30-4:30]

### [2:30] 가장 중요한 원칙
Peter가 말하는 AI 코딩 성공의 가장 중요한 원칙은 뭐냐면,
바로 "Loop를 닫는다"는 거예요.
[자막: "Close the Loop"]

이게 무슨 말이냐면요.

### [2:45] Loop 닫기란?
AI한테 코드를 짜라고 시켰으면,
그 AI가 스스로 자기가 짠 코드를 테스트하고 검증할 수 있어야 한다는 거예요.
[B-roll: AI가 코드 작성 → 테스트 → 디버깅하는 다이어그램]

Peter의 말을 직접 들어보면,
"The big secret is to always close the loop. It needs to be able to debug and test itself."
[자막: "루프를 항상 닫아야 해요. AI가 스스로 디버깅하고 테스트할 수 있어야 해요."]

### [3:10] 왜 중요한가?
왜 이게 중요하냐면요,
대부분의 사람들은 AI한테 코드 짜라고만 하고,
그걸 사람이 직접 돌려보고 테스트하잖아요.

근데 Peter는 달라요.
AI가 코드 짜면, AI가 직접 실행하고, 에러 나면 AI가 고치고,
테스트 통과할 때까지 계속 반복하게 해요.

### [3:35] 실제 작동 방식
예를 들어볼게요.
[B-roll: 시나리오 애니메이션]

1. AI한테 "WhatsApp 메시지 보내는 기능 만들어줘" 하면
2. AI가 코드 짜요
3. AI가 직접 실행해봐요
4. 에러 나면? AI가 에러 로그 읽고 코드 수정
5. 다시 실행
6. 통과할 때까지 반복

이게 바로 "Loop 닫기"예요.

### [4:00] 왜 대부분 못 하나?
근데 대부분의 사람들은 왜 이렇게 못 하냐면,
시스템을 그렇게 설계 안 해놨기 때문이에요.

AI한테 코드만 짜라고 하지,
"짜고 나서 직접 실행하고, 테스트하고, 로그 확인하고, 고쳐"라고 안 시킨다는 거죠.

### [4:15] 실천 팁
여러분이 바로 적용할 수 있는 팁을 드리자면,
Claude나 ChatGPT 같은 거 쓸 때,
"코드 짜줘"가 아니라
"코드 짜고, 실행하고, 테스트 통과할 때까지 계속 돌려봐"라고 하세요.

이게 Loop 닫기의 시작이에요.

## 섹션 3: 비밀 2 - 병렬 에이전트 운영 [4:30-6:30]

### [4:30] 600커밋의 진짜 비밀
자, 그럼 하루 600커밋은 어떻게 가능했을까요?
[자막: "600 커밋/일의 비밀"]

정답은, 에이전트를 여러 개 동시에 돌렸기 때문이에요.

### [4:45] 몇 개나 돌렸나?
Peter는 보통 5개에서 10개의 AI 에이전트를 동시에 운영한대요.
[B-roll: 5-10개 에이전트가 병렬로 작업하는 다이어그램]

"I use between 5 and 10 agents in parallel."
[자막: "저는 5-10개의 에이전트를 동시에 써요."]

### [5:00] 어떻게 운영하나?
작동 방식은 이래요.

한 에이전트한테 "A 기능 만들어줘" 하고 던져놓으면,
그 에이전트가 40분 동안 알아서 작업해요.

그 사이에 Peter는 뭐 하냐면,
다른 에이전트한테 "B 기능 만들어줘", "C 버그 고쳐줘" 하고 또 던지는 거예요.

### [5:20] 멀티태스킹의 정석
그러니까 Peter는 동시에 여러 가지 일을 하는 게 아니라,
여러 에이전트를 지휘하는 거예요.
[B-roll: 지휘자 이미지]

마치 오케스트라 지휘자처럼,
각 에이전트한테 역할 주고,
중간중간 확인하고,
문제 있으면 방향 잡아주고.

### [5:40] 어떤 도구 쓰나?
Peter는 Codex라는 도구를 주로 써요.
Claude Code보다 10배 느리다고 하는데,
거의 항상 정확해서 좋아한다고 해요.

그리고 CLI(커맨드 라인)를 선호해요.
왜냐면 여러 명령을 체이닝할 수 있거든요.

### [6:00] 일반인도 가능한가?
여러분은 "나는 개발자도 아닌데 가능해?" 생각할 수 있는데요,
원리는 똑같아요.

예를 들어, 블로그 글 5개를 써야 한다면,
AI 5개한테 각각 주제 하나씩 주고 동시에 돌리는 거예요.

한 AI가 30분 걸린다면,
순서대로 하면 150분, 병렬로 하면 30분이죠.

### [6:20] 핵심 마인드셋
핵심은 이거예요.
"내가 직접 하는 게 아니라, 누가 대신하게 만들 것인가?"
[자막: "직접 vs 지휘"]

Peter는 코더에서 지휘자로 역할을 바꾼 거예요.

## 섹션 4: 비밀 3 - 코드를 읽지 않는 이유 [6:30-8:30]

### [6:30] 충격적 고백
이제 가장 충격적인 부분이에요.

Peter는 인터뷰에서 이렇게 말했어요.
"Did I read all the code? No."
[자막: "모든 코드를 읽었냐고요? 아니요."]

### [6:45] 왜 안 읽나?
왜 안 읽을까요?
Peter의 설명을 들어보면,

"A lot of code really is just boring plumbing."
[자막: "대부분의 코드는 그냥 지루한 배관 작업이에요."]

무슨 말이냐면요.

### [7:00] 배관 작업이란?
코드의 80%는 사실 창의적인 게 아니래요.
[B-roll: 코드 예시 화면]

- 데이터 형식 변환
- API 연동
- 에러 핸들링
- 로깅
- 테스트 코드

이런 거 다 "배관 작업"이래요.
중요하긴 한데, 누가 짜든 비슷하게 나온다는 거죠.

### [7:20] 그럼 뭐가 중요한가?
Peter가 집중하는 건 뭐냐면,
시스템 아키텍처예요.
[자막: "아키텍처 > 코드"]

- 전체 시스템 설계
- 모듈 간 관계
- 데이터 흐름
- 확장성

이런 큰 그림이 중요하다는 거예요.

### [7:40] 코드 리뷰는?
그래서 Peter는 "Code reviews are dead"라고까지 말해요.
[자막: "코드 리뷰는 죽었다"]

대신 뭘 보냐면, 프롬프트를 봐요.

"I'm actually more interested in the prompts than in the code."
[자막: "저는 코드보다 프롬프트에 더 관심 있어요."]

### [8:00] PR = Prompt Request
그래서 Peter는 농담으로 이렇게 말해요.
"PR은 이제 Pull Request가 아니라 Prompt Request다."
[자막: "PR = Prompt Request"]

팀원들한테도 코드 올릴 때
"어떤 프롬프트로 이거 만들었는지도 같이 올려달라"고 한대요.

### [8:15] 논란의 여지
물론 이건 논란이 있을 수 있어요.

코드를 안 읽으면 보안 이슈나 버그를 놓칠 수도 있잖아요.

하지만 Peter의 관점은 이거예요.
"AI가 테스트를 다 돌리고, 내가 시스템 아키텍처를 잘 잡으면,
한 줄 한 줄 읽는 것보다 더 안전하다."

## 섹션 5: 새로운 정체성 - "Builder" [8:30-9:30]

### [8:30] Coder vs Builder
그럼 이제 Peter는 자신을 뭐라고 부를까요?

Coder? 아니요.
Architect? 그것도 아니래요.

### [8:45] Builder의 등장
Peter가 좋아하는 단어는 "Builder"예요.
[자막: "Builder"]

"I wouldn't say architecture. I like the word builder."
[자막: "아키텍처라고 말하지 않아요. 저는 빌더라는 단어가 좋아요."]

### [9:00] Builder란?
Builder는 뭐냐면요,
[B-roll: Builder 개념 다이어그램]

1. 제품 비전을 가진 사람
2. 시스템 전체를 설계하는 사람
3. AI를 도구처럼 쓰는 사람
4. 취향과 철학을 제품에 담는 사람

단순히 코드 짜는 사람이 아니라,
뭔가를 만들어내는 사람이라는 거예요.

### [9:20] 우리도 Builder
재밌는 건, 이제 누구나 Builder가 될 수 있다는 거예요.

예전엔 코딩 못 하면 제품 못 만들었잖아요.
근데 지금은 비전만 있으면, AI한테 만들게 할 수 있어요.

## 아웃트로 [9:30-11:00]

### 핵심 요약 [9:30-10:00]
오늘 내용 정리해드릴게요.

**첫째, Loop를 닫으세요.**
AI한테 코드 짜라고만 하지 말고,
테스트하고 검증할 때까지 스스로 돌리게 하세요.

**둘째, 병렬로 운영하세요.**
한 번에 하나씩 하지 말고,

**셋째, 큰 그림을 보세요.**
한 줄 한 줄 코드보다,
시스템 아키텍처와 비전이 더 중요해요.

**넷째, 당신은 Builder예요.**
코더가 아니라 만드는 사람.
AI는 도구일 뿐이에요.

### 솔직한 평가 [10:00-10:30]
솔직히 말씀드리면,
Peter의 방식이 모든 사람한테 맞는 건 아니에요.

특히 엔터프라이즈 환경이나,
보안이 중요한 곳에서는 코드를 꼼꼼히 봐야 할 수도 있어요.

하지만 1인 창업가나,
빠르게 프로토타입 만들어야 하는 사람들한테는
정말 유용한 접근법이에요.

저도 이 방식을 제 프로젝트에 적용하기 시작했는데,
생산성이 확실히 달라지더라고요.

구독과 알림 설정도 부탁드려요.
