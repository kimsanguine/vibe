# AI 에이전트로 하루 600커밋 하는 개발자의 5가지 생산성 원칙

혼자서 모든 걸 해야 하는 개발자에게 시간은 가장 소중한 자원이에요. 제품 아이디어는 많은데, 구현할 시간은 턱없이 부족하죠. 그런데 하루에 600번의 커밋을 올리면서도 "대부분의 코드를 읽지 않는다"고 말하는 개발자가 있어요.

Peter Steinberger는 10억 개 이상의 디바이스에서 사용되는 PSPDF Kit를 만든 개발자예요. 13년간 회사를 운영하다 번아웃으로 3년간 쉬었다가, 2024년 AI 에이전트 시대에 복귀했어요. 그가 공개한 AI 에이전트 활용법은 1인 개발자나 소규모 팀에게 큰 인사이트를 줘요.

이 글에서는 Peter의 실제 작업 방식을 바탕으로, AI 에이전트로 개발 생산성을 10배 올리는 5가지 원칙을 소개할게요. 각 원칙마다 구체적인 적용 방법과 팁도 함께 담았어요.

## Peter Steinberger는 누구인가요?

Peter는 오스트리아 출신 개발자로, iOS 생태계에서 유명한 PDF 프레임워크 PSPDF Kit을 만들었어요. 10억 개가 넘는 디바이스에서 그가 만든 기술이 사용되고 있죠.

13년간 회사를 운영하며 극도로 높은 집중력으로 일했지만, 결국 번아웃으로 3년간 기술에서 완전히 떠났어요. 2024년 AI 에이전트 시대가 열리면서 복귀를 결심했고, 현재는 Claudebot이라는 AI 개인 비서 프로젝트를 진행 중이에요.

이 프로젝트는 출시 1주일 만에 GitHub 스타가 100개에서 3,300개로 급증했어요. WhatsApp, Discord, Slack, iMessage 등 멀티 플랫폼을 지원하고, 사용자의 컴퓨터에 풀 액세스해서 카메라, 홈 자동화, 음악까지 제어할 수 있어요.

## 하루 600커밋의 비밀: 병렬 에이전트 운영

Peter는 어느 날 하루에 600번의 커밋을 올렸어요. 사람이 손으로 치는 게 불가능한 숫자죠. 비밀은 바로 **병렬 에이전트 운영**이에요.

그는 보통 5개에서 10개의 AI 에이전트를 동시에 돌려요. 한 에이전트가 40분 동안 작업하는 동안, 다른 에이전트에게 다른 작업을 지시하는 거예요. 한 명이 밤낮으로 일하는 게 아니라, 작은 팀을 운영하는 것처럼요.

"한 에이전트가 작업하는 동안 기다리지 않아요. 바로 다른 에이전트에게 다음 작업을 맡기죠. 이렇게 하면 하루 종일 뭔가가 계속 만들어져요."

실제로 적용하려면 이렇게 해보세요:
- 작업을 독립적인 단위로 쪼개세요 (기능별, 모듈별)
- 각 에이전트에게 명확한 범위를 지정하세요
- 동시에 진행할 수 있는 작업들을 미리 리스트업하세요
- 한 에이전트가 막히면 바로 다음 작업으로 넘어가세요

병렬 운영의 핵심은 "기다리지 않는 것"이에요. 한 곳에서 막히면 다른 곳을 진행하면서, 전체 프로젝트가 계속 앞으로 나아가게 만드는 거죠.

## 원칙 1: "Loop 닫기" - AI 코딩 성공의 핵심

Peter가 강조하는 가장 중요한 원칙은 "Loop를 닫는 것"이에요. 이게 뭘까요?

AI가 코드를 작성하면, 반드시 스스로 테스트하고 검증할 수 있어야 해요. 사람이 개입해서 "이게 맞나? 저게 맞나?" 확인하는 게 아니라, AI 스스로 자기 작업을 디버깅하고 테스트할 수 있게 시스템을 설계하는 거예요.

"Loop를 닫는 게 가장 큰 비밀이에요. AI가 스스로 디버깅하고 테스트할 수 있어야 해요."

구체적으로 어떻게 할까요?

**1. 테스트 자동화 설정**
- 유닛 테스트, 통합 테스트를 작성해두세요
- AI가 코드를 수정할 때마다 자동으로 테스트가 돌아가게 하세요
- 테스트가 실패하면 AI가 그 결과를 보고 다시 수정하게 하세요

**2. 즉각적인 피드백 시스템**
- 에러 로그를 AI가 바로 읽을 수 있게 하세요
- 빌드 실패 메시지를 AI에게 전달하세요
- AI가 자기 실수를 직접 고칠 수 있는 환경을 만드세요

**3. 로컬 환경에서 검증**
- Peter는 CI/CD 파이프라인보다 로컬 테스트를 선호해요
- 에이전트가 이미 로컬에서 충분히 테스트했기 때문이죠
- 빠른 피드백 루프가 생산성의 핵심이에요

Loop를 닫으면 사람의 개입이 최소화돼요. AI가 자기 작업을 책임지고 완성할 수 있게 되는 거죠. 처음 설정은 시간이 걸리지만, 한 번 만들어두면 계속 효과를 봐요.

## 원칙 2: PR은 이제 "Prompt Request"

코드 리뷰 문화가 많이 바뀌고 있어요. Peter는 이제 풀 리퀘스트(PR)의 코드보다 **프롬프트**에 더 관심이 많아요.

"코드보다 프롬프트가 더 궁금해요. 사람들에게 프롬프트를 꼭 추가해달라고 요청해요."

왜 그럴까요? AI가 만든 코드는 대부분 "지루한 배관 작업"이에요. 데이터를 변환하고, API를 연동하고, 에러를 처리하는 반복적인 작업이죠. 중요한 건 **"무엇을 만들려고 했나"** 하는 의도예요.

실제로 이렇게 적용해보세요:

**프롬프트 문서화**
```markdown
## 프롬프트
"사용자 인증 API를 구현해줘. JWT 토큰 방식이고,
리프레시 토큰도 지원해야 해. 에러 처리는 표준 HTTP 상태 코드 사용."

## 결과
- /auth/login 엔드포인트 추가
- JWT 미들웨어 구현
- 리프레시 토큰 로직 추가
```

프롬프트를 함께 기록하면 좋은 점:
- 나중에 비슷한 작업할 때 재사용 가능
- 다른 팀원이 맥락을 바로 이해해요
- 프롬프트를 개선하는 방법을 배울 수 있어요
- 코드 한 줄 한 줄 리뷰하는 시간 절약

Peter는 "PR"을 "Prompt Request"로 부르기도 해요. 코드를 리뷰하는 게 아니라, 어떤 프롬프트로 어떤 문제를 해결했는지 공유하는 거죠.

## 원칙 3: 코드를 읽지 않는다고요?

Peter의 가장 도발적인 발언은 이거예요: "대부분의 코드를 읽지 않고 배포해요."

충격적으로 들리지만, 그에게는 명확한 이유가 있어요. 대부분의 코드는 사실 **"지루한 배관 작업"**이에요. 데이터 형식을 변환하고, API를 호출하고, 에러를 잡는 반복적인 일이죠.

"코드를 다 읽었냐고요? 아니요. 많은 코드가 정말 지루한 배관 작업이거든요."

그럼 Peter는 무엇에 집중할까요?

**1. 시스템 아키텍처**
- 전체 구조가 어떻게 설계되는지
- 모듈 간 의존성은 어떤지
- 확장 가능한 구조인지

**2. 테스트 결과**
- 모든 테스트가 통과하는지
- 엣지 케이스가 제대로 처리되는지
- 성능 이슈는 없는지

**3. 제품 동작**
- 실제로 사용해봤을 때 예상대로 작동하는지
- 사용자 경험은 괜찮은지
- 버그나 이상한 동작은 없는지

이 접근법이 가능한 이유는 원칙 1 "Loop 닫기" 덕분이에요. AI가 스스로 테스트하고 검증하니까, 사람이 한 줄 한 줄 읽지 않아도 되는 거예요.

물론 모든 코드를 안 읽는 건 아니에요. 핵심 비즈니스 로직이나 보안이 중요한 부분, 복잡한 알고리즘은 여전히 사람이 확인해요. 하지만 반복적인 CRUD 작업이나 간단한 기능은 테스트만 통과하면 OK예요.

## 원칙 4: "Builder"가 되세요

Peter는 스스로를 더 이상 "coder"나 "architect"라고 부르지 않아요. 대신 **"builder"**라는 단어를 선호해요.

"아키텍트라고 하지 않을래요. 저는 '빌더'라는 단어가 좋아요."

Builder가 뭐가 다를까요?

**Coder vs Builder**
- Coder: 사양서를 받아 코드로 구현
- Builder: 제품 비전을 갖고, 취향을 적용하며, 전체를 책임짐

Builder는 이런 일을 해요:
- 무엇을 만들지 결정
- 시스템 전체를 설계
- AI 에이전트들을 지휘
- 제품의 방향을 제시
- 사용자 경험을 다듬음

Peter가 Codex를 선호하는 이유도 이것 때문이에요. Claude Code보다 10배 느리지만, 거의 항상 정확해요. 그는 모델과 "대화"하며 설계하고, 바로 빌드 명령을 내리지 않아요. 충분히 논의하고 방향을 정한 다음 작업을 시작하죠.

Builder 마인드셋으로 일하는 방법:

**1. 제품 비전부터 시작**
```
"이 기능이 왜 필요한가?"
"사용자에게 어떤 가치를 줄까?"
"어떻게 하면 더 나은 경험을 만들까?"
```

**2. AI와 대화하며 설계**
- 바로 코딩을 시키지 말고, 먼저 아이디어를 논의하세요
- 여러 접근법을 물어보고 비교하세요
- 장단점을 듣고 최선의 방법을 선택하세요

**3. 디테일에 취향 적용**
- AI가 만든 결과물에 나만의 스타일 추가
- 사용자 경험의 작은 부분까지 신경 쓰기
- 제품에 개성 불어넣기

Builder는 단순히 코드를 짜는 게 아니라, 제품을 만들어요. AI 시대에는 이런 역할이 더 중요해져요.

## 원칙 5: CLI와 체이닝을 활용하세요

Peter는 MCP(Model Context Protocol) 같은 복잡한 도구보다 **CLI(커맨드라인 인터페이스)**를 선호해요. 간단하고, 체이닝이 가능하기 때문이죠.

CLI가 왜 효율적일까요?

**1. 간단함**
- 복잡한 설정 없이 바로 사용
- 표준 입출력으로 데이터 주고받기
- 디버깅이 쉬움

**2. 체이닝 가능**
```bash
# 여러 도구를 파이프로 연결
cat data.json | jq '.users[]' | grep "active" | wc -l
```

AI 에이전트도 이렇게 작업을 연결할 수 있어요:
- 에이전트 A가 데이터 수집
- 에이전트 B가 데이터 가공
- 에이전트 C가 결과 출력

**3. 재사용성**
- 한 번 만든 스크립트를 계속 쓸 수 있어요
- 다른 프로젝트에도 적용 가능
- 팀원들과 공유하기 쉬워요

Peter의 또 다른 팁:
- **테스트와 문서는 100% AI에게 위임**하세요
- 단, 테스트가 잘 작동하는지는 확인하세요
- 문서는 코드에서 자동으로 생성되게 하세요

## 실전 적용: 내일부터 시작하는 방법

이론은 충분해요. 지금 바로 시작할 수 있는 구체적인 방법을 정리할게요.

**1단계: 환경 설정 (1-2시간)**
- Claude Code 또는 Codex 설치
- 테스트 자동화 설정 (pytest, jest 등)
- Git hook으로 커밋 전 테스트 실행하게 설정

**2단계: 작은 작업부터 시도 (첫 주)**
- 테스트 코드 작성을 AI에게 맡기기
- 간단한 CRUD API AI로 만들기
- 프롬프트를 커밋 메시지에 기록하기

**3단계: 병렬 작업 연습 (2-3주차)**
- 독립적인 기능 2-3개 동시 진행
- 한 에이전트가 작업하는 동안 다른 작업 준비
- Loop가 잘 닫히는지 확인하며 조정

**주의할 점**
- 처음부터 완벽하게 하려고 하지 마세요
- 실패해도 괜찮아요. AI 코딩은 시행착오가 필요해요
- 중요한 비즈니스 로직은 여전히 사람이 리뷰하세요
- 보안 관련 코드는 반드시 확인하세요

**실패 사례에서 배우기**
- 테스트 없이 배포했다가 버그 발견
- 프롬프트가 모호해서 원하는 결과 못 얻음
- 에이전트끼리 충돌하는 코드 작성

이런 실수들은 자연스러워요. 겪으면서 Loop 닫기를 개선하고, 프롬프트 작성 실력이 늘어요.

## 마무리: 개발자의 역할이 바뀌고 있어요

Peter Steinberger의 이야기가 시사하는 건 단순히 "AI로 빠르게 코딩하기"가 아니에요. **개발자의 역할 자체가 바뀌고 있다**는 거예요.

핵심을 다시 정리하면:
1. **Loop 닫기**: AI가 스스로 테스트하고 검증하게 시스템 설계
2. **병렬 운영**: 여러 에이전트로 동시에 작업 진행
3. **프롬프트 중심**: 코드보다 의도가 중요
4. **아키텍처 집중**: 한 줄 한 줄보다 전체 구조
5. **Builder 마인드**: 제품 비전을 갖고 AI를 지휘

AI 시대의 개발자는 더 이상 코드를 치는 사람이 아니에요. 제품을 설계하고, 시스템을 구성하고, AI 팀을 이끄는 Builder예요.

Peter는 번아웃 후 3년 만에 복귀해서 1주일 만에 수천 명이 관심 갖는 프로젝트를 만들었어요. 당신도 할 수 있어요. 오늘부터 작은 작업 하나를 AI 에이전트에게 맡겨보세요. Loop를 닫고, 테스트를 자동화하고, 프롬프트를 기록하세요.

그리고 한 걸음 물러서서, 당신이 정말 만들고 싶은 제품이 무엇인지 생각해보세요. 이제 구현은 AI가 도와줄 거예요. 당신은 Builder로서 방향을 제시하면 돼요.
